01 How it works

One tunnel, three transports.

Velox mimics normal HTTPS traffic. Your client connects to the server using WebSocket, HTTP/2, or QUIC. The server forwards traffic to wherever you need — invisible to any proxy or firewall that allows web traffic.

Your machine
velox client
local socks5 / tcp / udp
WebSocket HTTP/2 QUIC
TLS · disguised as HTTPS
Your VPS
velox server
wss://0.0.0.0:443
TCP · UDP · Unix
Destination
any host:port
google.com · ssh · db · VPN
01

Start server on your VPS

velox server wss://[::]:443

Single binary. Self-signed cert included. Bring your own cert for production.

02

Connect from your machine

velox client -L socks5://127.0.0.1:8888 wss://yourserver:443

Sets up a local SOCKS5 proxy. Point any app at it.

03

Traffic flows freely

curl -x socks5h://127.0.0.1:8888 http://google.com

DNS, TCP, UDP — all tunneled. Firewall sees only TLS on port 443.

02 Transports

03 Features

Everything you need, nothing you don't.

↔

Forward & reverse tunnels

Use -L to forward local traffic to remote hosts. Use -R to expose local services on the server.

◇

Dynamic proxying

SOCKS5, HTTP proxy, and transparent proxy (tproxy, Linux only) for routing any application's traffic.

▭

TCP · UDP · Unix · stdio

Tunnel raw TCP, UDP with configurable idle timeout, Unix sockets, and stdio for SSH ProxyCommand.

☈

mTLS & access control

Authenticate clients with certificates. Restrict by path prefix, destination, or a hot-reloading YAML rules file.

↻

Auto cert reload

TLS certificates, keys, and CA bundles are watched and reloaded without dropping connections.

▤

Connection pool

Keep idle connections with --connection-min-idle to eliminate TLS handshake latency for SOCKS5 proxying.

◎

Corporate proxy support

Connect through an HTTP proxy with -p http://proxy:8080 before reaching the velox server.

◌

ECH & SNI override

Override TLS SNI to impersonate any domain. Enable ECH to hide the SNI from deep packet inspection.

⬤

Single static binary

No runtime, no dependencies. One binary for Linux, macOS, Windows, ARM, Android. Drop and run.

04 Quick start

Common setups.

SOCKS5 proxy for browser/curl
Server
velox server wss://[::]:8080
Client
velox client -L socks5://127.0.0.1:8888 \ --connection-min-idle 5 \ wss://myserver:8080
Test
curl -x socks5h://127.0.0.1:8888 http://google.com
SSH via ProxyCommand
Server
velox server wss://[::]:443 \ --restrict-to 127.0.0.1:22
Client
ssh -o ProxyCommand="velox client \ --log-lvl=off \ -L stdio://%h:%p \ ws://myserver:443" my-server
WireGuard over QUIC Lowest latency
Server
velox server wss://[::]:8080 \ --quic-bind 0.0.0.0:8443 \ --restrict-to localhost:51820
Client
velox client \ -L 'udp://51820:localhost:51820?timeout_sec=0' \ quic://myserver:8443
Through a corporate HTTP proxy
Server
velox server wss://[::]:443
Client
velox client \ -L tcp://9999:127.0.0.1:22 \ -p http://corp-proxy:8080 \ wss://myserver:443
Then
ssh -p 9999 user@127.0.0.1
Reverse tunnel — expose local service
Client
velox client \ -R 'tcp://[::]:8000:localhost:8000' \ wss://myserver:443
From server
curl http://localhost:8000 # hits your local machine
Transparent proxy — proxy any app Linux only
Client
sudo velox client \ -L 'tproxy+tcp://1080' \ -L 'tproxy+udp://1080' \ wss://myserver:443
Proxy any process
cproxy --port 1080 --mode tproxy \ -- curl https://google.com
